Overview

4 个观察面

制度面

`manifest.json`

定义部门、产物、路由和约束。

状态面

`state.json` + `runtime/control.json`

记录阶段、负责人、写入范围和阻塞。

证据面

评审 / 审计 / 事件流

人类可读文档加机器可读轨迹。

执行面

调度器与驱动

负责选任务、加租约、执行和写回。

Flow

主流程

皇帝
提出需求 只描述目标和约束，不亲自推进每一步。
↓
中书
规格起草 把需求整理成规格、验收标准、非目标和上下文。
↓
门下
需求评审 只给通过或驳回结论。驳回会回流到规格补写。
↓
吏部
任务拆分 拆工作流、依赖和责任归属。
礼部
原型设计 先固化页面结构与交互，再让后续设计继续。
门下
UI 评审 冻结原型方向，驳回则回流到礼部。
↓
工部
API / 设计 / 构建 产出契约、数据模型、迁移与真实交付。
刑部
规则 / 测试 / 验证 定义门禁、用例和验证结果。
门下
API / 测试评审 工部与刑部都要经过评审门禁，才能进入下一步。
↓
御史台
审计复核 治理偏差回退到规划，执行偏差回退到构建。
↓
尚书
完成归档 状态、事件、评审和审计证据全部闭环。

关键规则

门下只评审，御史台独立审计，尚书唯一总控。
