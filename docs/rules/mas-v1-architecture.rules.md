# mas-v1-architecture Rules

## 命中规则清单

### RULE-001
- 来源: 用户确认的 MAS V1 设计方向
- 命中条件: 定义整体执行流程时
- 约束内容: 必须体现“需求澄清 -> 原型优先 -> API 跟随原型 -> 测试前置 -> 构建验证 -> 独立审计”的主链路
- 严重级别: high
- 检查方式: review

### RULE-002
- 来源: 用户确认的组织方式
- 命中条件: 设计多 Agent 协作方式时
- 约束内容: 采用“单总控编排 + 有限平级协商”，不能把全流程设计成自由协商网络
- 严重级别: high
- 检查方式: review

### RULE-003
- 来源: 用户确认的产品流程
- 命中条件: 定义需求、UI、API 相关阶段时
- 约束内容: 原型设计必须优先于 API 设计，API 必须跟随原型产出
- 严重级别: high
- 检查方式: review

### RULE-004
- 来源: 用户确认的开发方式
- 命中条件: 定义实现阶段时
- 约束内容: 必须先冻结 API 契约，再允许前端、后端、数据库并行推进
- 严重级别: high
- 检查方式: review

### RULE-005
- 来源: 用户确认的测试策略
- 命中条件: 定义测试与构建阶段时
- 约束内容: 测试必须先于代码开发，并作为进入构建阶段的门禁
- 严重级别: high
- 检查方式: review

### RULE-006
- 来源: 用户确认的职责划分
- 命中条件: 设计数据库职责时
- 约束内容: 数据库设计与数据库迁移必须归属于同一实现部门
- 严重级别: medium
- 检查方式: review

### RULE-007
- 来源: 用户确认的范围边界
- 命中条件: 定义 V1 终点时
- 约束内容: V1 只做到构建、验证和审计完成，不纳入部署发布
- 严重级别: high
- 检查方式: review

### RULE-008
- 来源: 三省六部映射设计
- 命中条件: 编写角色职责映射时
- 约束内容: 必须明确中书省、门下省、尚书省、吏部、礼部、工部、刑部、御史台的对应 Agent 职责
- 严重级别: medium
- 检查方式: review

### RULE-009
- 来源: 可落地性要求
- 命中条件: 设计目录结构与状态机时
- 约束内容: 必须给出可执行的任务目录结构和严格状态机，而不是只停留在概念层
- 严重级别: high
- 检查方式: review

### RULE-010
- 来源: schema 落地要求
- 命中条件: 定义 `state.json` 与 `manifest.json` 时
- 约束内容: 必须提供字段定义、枚举约束和示例，且与主文档中的流程机制一致
- 严重级别: high
- 检查方式: schema 校验 + review

### RULE-011
- 来源: 权限隔离要求
- 命中条件: 设计部门权限时
- 约束内容: 各部门只写自己的授权区域，不能让审计、评审、实现混写同一责任边界
- 严重级别: medium
- 检查方式: review

### RULE-012
- 来源: manifest 部门一致性修复
- 命中条件: 设计 `manifest.departments` 时
- 约束内容: 必须使用 `libu_task_breakdown` 与 `libu_prototype` 两个独立 key，不能再混用单一 `libu`
- 严重级别: high
- 检查方式: schema 校验 + review

### RULE-013
- 来源: 文档恢复要求
- 命中条件: 修复 MAS V1 文档内容时
- 约束内容: 主文档和配套 workflow 文档必须恢复为正常中文可读内容，不能残留真实 `?` 占位导致语义丢失
- 严重级别: high
- 检查方式: review + 文件检查
